[PonsLink 기술 회고] WebRTC 제품을 상태 중심으로 다시 설계하기

PonsLink를 다시 돌아보면 핵심은 WebRTC를 붙였다는 사실보다, 끊어지는 연결과 늦게 도착하는 상태를 제품이 어떻게 견디게 만들었는지에 있다.

이번 글에서는 PonsLink 프로젝트에서 사용한 기술 스택과 구조를 정리한다. 특히 React/TypeScript 프론트엔드, WebRTC mesh, Socket.IO signaling, Node.js/Express API, PostgreSQL/Redis 기반 상태 저장, 요청 데스크와 결제 흐름이 어떤 책임으로 나뉘었는지를 중심으로 본다.

1. 실시간 제품의 기준선: 화면보다 상태
PonsLink는 겉으로 보면 실시간 통화 제품이다. 하지만 구현을 따라가면 먼저 해결해야 했던 것은 카메라 화면이 아니라 상태였다. 누가 방에 들어왔는지, 어떤 peer가 아직 살아 있는지, 어떤 메시지는 반드시 살아야 하는지, 어떤 메시지는 최신 값만 있으면 되는지부터 정해야 했다.

2. PonsLink 기술 스택
Frontend: React, TypeScript, Vite, Zustand, TanStack React Query, Radix UI, Tailwind, Konva/react-konva.
Realtime: WebRTC Mesh, Socket.IO, DataChannel, ICE/STUN/TURN fallback, replay/resume sequence.
Backend: Node.js, Express, TypeScript, Socket.IO server, PostgreSQL, Redis, S3 presigned URL, Nodemailer.
Product flow: Public Desk request, reservation, session-access, Polar, LemonSqueezy fallback, checkout idempotency.
Collaboration: file chunk transfer, IndexedDB, Comlink worker, whiteboard, live chat, PonsCast, CoWatch.

3. 실시간 동작 원리: Join, Signal, Mesh, Replay
방 입장은 join token을 확보하고 socket 연결을 맺은 뒤 room state를 받는 순서로 움직인다. signaling broker는 offer, answer, ice-candidate, renegotiate, bye를 정규화하고 payload와 session을 검증한다. peer 연결은 결정적 initiator 규칙으로 glare를 줄이고, replay sequence로 reconnect 이후 놓친 이벤트를 따라잡는다.

4. DataChannel 정책: 모든 메시지를 같은 방식으로 보내지 않는다
채팅과 제어 메시지는 reliable 성격이 강하다. 화이트보드 커서나 diagnostics는 최신 상태만 살아 있어도 된다. 파일은 chunk, ack, bufferedAmount, resume이 필요하다. 그래서 PonsLink는 reliable queue와 latest-wins queue를 나누고, congestion 상황에서 버릴 수 있는 메시지를 분리했다.

5. Request-first 전환
후반의 PonsLink는 room-first에서 request-first로 이동했다. 사용자는 먼저 요청을 보내고, host가 수락하면 reservation이 만들어지고, join window가 열릴 때 방 입장이 가능해진다. requestStateModel은 요청, 예약, 메일, 접근 권한을 접어서 submitted, reviewing, accepted, email_delivered, join_window_open, connecting, completed 같은 UX 상태로 보여준다.

6. 적용 원칙
첫째, 검증은 방 안이 아니라 방 앞에서 끝낸다. 둘째, 메시지마다 생존 등급을 나눈다. 셋째, 내부 상태를 그대로 노출하지 말고 사용자 문장으로 접는다.

마치며
PonsLink의 기술 선택은 라이브러리 목록보다 책임 배치가 더 중요했다. WebRTC는 연결을 만들었고, Socket.IO는 협상과 replay를 맡았고, Zustand와 React Query는 클라이언트 상태를 나눴고, PostgreSQL/Redis는 요청과 세션의 기준선을 만들었다. 좋은 실시간 제품은 더 많은 이벤트를 보내는 제품이 아니라, 잃어도 되는 것과 잃으면 안 되는 것을 정확히 구분하는 제품에 가깝다.
