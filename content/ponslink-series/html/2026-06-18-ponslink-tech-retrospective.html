PonsLink를 다시 돌아보면 핵심은 WebRTC를 붙였다는 사실보다, 끊어지는 연결과 늦게 도착하는 상태를 제품이 어떻게 견디게 만들었는지에 있다.

이번 글에서는 PonsLink 프로젝트에서 사용한 기술 스택과 구조를 정리한다. 특히 React/TypeScript 프론트엔드, WebRTC mesh, Socket.IO signaling, Node.js/Express API, PostgreSQL/Redis 기반 상태 저장, 요청 데스크와 결제 흐름이 어떤 책임으로 나뉘었는지를 중심으로 본다.

1. 실시간 제품의 기준선: 화면보다 상태

PonsLink는 겉으로 보면 실시간 통화 제품이다. 하지만 구현을 따라가면 먼저 해결해야 했던 것은 카메라 화면이 아니라 상태였다. 누가 방에 들어왔는지, 어떤 peer가 아직 살아 있는지, 어떤 메시지는 반드시 살아야 하는지, 어떤 메시지는 최신 값만 있으면 되는지부터 정해야 했다.

구분처음 생각하기 쉬운 문제PonsLink가 실제로 붙잡은 문제
방 입장URL을 열면 된다join token, capacity, participant store, replay 기준선이 필요하다
미디어카메라와 마이크를 연결하면 된다ICE 준비, TURN fallback, peer initiator 규칙이 필요하다
협업 메시지DataChannel로 보내면 된다reliable, latest-wins, chunked transfer를 나눠야 한다
제품 흐름방을 만들고 초대하면 된다요청, 수락, 예약, 입장 가능 시간이 먼저 정해져야 한다

이 관점이 중요했다. 실시간 제품에서 화면은 결과이고, 상태는 원인이다. 상태 기준선이 흔들리면 WebRTC 연결이 살아 있어도 사용자는 제품이 불안정하다고 느낀다.

3. 실시간 동작 원리: Join, Signal, Mesh, Replay

방 입장은 단일 클릭처럼 보이지만 내부에서는 단계가 나뉜다. join token을 확보하고, Socket.IO 연결을 맺고, broker가 세션을 검증하고, room state를 받은 뒤 peer connection을 만든다.

단계주요 로직이 단계가 필요한 이유
Joinjoin token 확보, room capacity 확인, participant 등록방에 들어올 수 있는 사용자인지 먼저 정한다
Signaloffer/answer/ice-candidate 정규화, payload 제한, room/session 검증잘못된 signaling이 WebRTC 협상까지 들어오지 않게 막는다
MeshlocalUserId와 peerId 비교로 initiator 결정, ICE 준비 대기, STUN/TURN fallback동시 offer 충돌을 줄이고 첫 연결 시간을 관리한다
Replayroom replay sequence 저장, reconnect 후 resume-room, ack재접속 뒤 놓친 room event를 따라잡는다

이 구조에서 Socket.IO는 단순 메시지 통로가 아니다. 방의 출입문, 협상 중계자, replay 기준선 역할을 동시에 한다.

4. DataChannel 정책: 모든 메시지를 같은 방식으로 보내지 않는다

PonsLink 방 안에는 채팅, 화이트보드, 파일 전송, 자막, CoWatch, PonsCast가 같이 흐른다. 이 메시지를 모두 reliable로 보내면 지연이 커지고, 모두 latest-wins로 보내면 중요한 데이터가 사라진다. 그래서 메시지 생존 등급을 나눴다.

메시지 유형정책이유
control/textordered reliable빠지면 대화와 상태가 깨진다
whiteboard cursor/media/diagnosticslatest-wins, TTL, congestion drop오래된 값보다 최신 값 하나가 더 중요하다
file transferchunk, ack, bufferedAmount, resume순서와 누락 복구가 필요하다

파일 전송에서는 peer가 허용하는 max message size에서 header와 safety margin을 빼고 chunk size를 계산한다. 전송 상태는 IndexedDB에 pending transfer로 남기고, worker가 chunk를 준비하면 main thread가 bufferedAmount를 보고 잠깐 쉬게 한다.

5. Request-first 전환

후반의 PonsLink는 room-first에서 request-first로 이동했다. 사용자는 먼저 요청을 보내고, host가 수락하면 reservation이 만들어지고, join window가 열릴 때 방 입장이 가능해진다. 방은 제품 흐름의 시작점이 아니라 마지막 실행 지점이 됐다.

내부 리소스대표 상태사용자에게 보이는 의미
Requestsubmitted, reviewing, accepted, declined요청이 접수됐는지, 검토 중인지, 거절됐는지
Reservationreservation_created, completed, expired예약이 만들어졌는지, 끝났는지, 만료됐는지
Mailemail_delivered입장 안내가 전달됐는지
Session accesswaiting, allowed, payment_required, expired지금 들어갈 수 있는지, 결제가 필요한지

requestStateModel의 역할은 raw status를 그대로 보여주는 것이 아니라 여러 리소스를 하나의 UX 상태로 접는 것이다. 이 덕분에 사용자는 복잡한 내부 상태 대신 “요청 완료”, “입장 대기”, “연결 가능” 같은 문장으로 상황을 이해한다.

6. 적용 원칙

PonsLink를 다시 정리하면 남는 원칙은 세 가지다.

원칙구현에서 보인 형태제품 효과
Bounded boundarysignal payload, room token, access token, rate limit을 앞단에서 검증잘못된 연결이 방 안으로 들어오기 전에 차단
Message survival classreliable queue, latest-wins queue, TTL, congestion drop반드시 살아야 하는 메시지와 버려도 되는 메시지를 분리
State foldingrequest + reservation + mail + session-access → UX state복잡한 내부 상태를 사용자가 이해할 한 문장으로 변환

7. 마치며

지금까지의 내용은 세 가지 메시지로 요약된다.

  1. PonsLink의 핵심은 WebRTC 자체가 아니라 상태 기준선이다.
  2. 실시간 메시지는 생존 등급을 나눠야 한다. 모든 메시지를 같은 방식으로 보내면 제품 경험이 흔들린다.
  3. 방은 제품 흐름의 시작점이 아니라 마지막 실행 지점이 될 수 있다. 요청, 예약, 결제, 입장 가능 시간이 먼저 정해질 때 실시간 방이 더 안정적인 제품 흐름 안에 들어간다.
좋은 실시간 제품은 더 많은 이벤트를 보내는 제품이 아니라, 잃어도 되는 것과 잃으면 안 되는 것을 정확히 구분하는 제품에 가깝다.
