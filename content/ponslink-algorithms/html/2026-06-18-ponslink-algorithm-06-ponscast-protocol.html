PonsLink에는 "PonsCast"라는 기능이 있다. 회의 중에 녹화된 영상이나 PDF 자료를 모든 참여자에게 동시에 보여주는 기능이다. 표준 WebRTC라면 화면 공유를 쓰겠지만, PonsLink는 다른 선택을 했다. DataChannel 위에 직접 바이너리 프레임을 만들어 보낸다.

💡 핵심 — PonsCast는 화면 공유 대신 DataChannel로 미디어 프레임을 직접 전송한다. 인코딩이나 트랜스코딩 없이, 발신자가 가진 비디오 포맷 그대로 수신자에게 전달된다.

📡 왜 화면 공유를 안 쓰는가

화면 공유는 편하지만 제약이 많다. 브라우저마다 captureScreen API 지원 범위가 다르고, iOS Safari는 아예 화면 공유가 불가능하다. 권한 팝업이 뜨고, 사용자가 "화면 공유 중"이라는 걸 인식해야 하며, 공유하는 쪽의 전체 화면이 노출된다.

PonsCast는 이 문제를 역발상으로 풀었다. "화면을 공유하는 대신, 미디어 데이터를 데이터 채널로 보낸다." 발신자가 영상을 프레임 단위로 쪼개서 DataChannel로 보내면, 수신자는 그것을 받아 MediaSource API로 재생한다. 화면 공유 권한도, 화면 노출도 없다.

🏗️ 전체 파이프라인

발신자 쪽 흐름: 비디오 요소에서 미디어를 캡처하고 → 청크 단위로 분할하고 → 13바이트 프레임 헤더를 붙이고 → 토큰 버킷 레이트 리밋으로 속도를 조절하고 → DataChannel로 보낸다.

수신자 쪽 흐름: DataChannel에서 바이너리 데이터를 받고 → 프레임 헤더를 파싱하고 → 시퀀스 번호 순서대로 jitter buffer에 넣고 → MediaSource + SourceBuffer에 appendBuffer로 추가하고 → 비디오 요소에서 재생한다.

이 파이프라인의 핵심은 "미디어를 코드화하지 않고, 있는 그대로 잘라서 보낸다"는 점이다. 인코딩이나 트랜스코딩이 없다. 발신자가 가진 비디오 포맷 그대로 수신자에게 전달된다.

📦 프레임 프로토콜: 13바이트에 담긴 것

PonsCast 프레임은 고정 크기 헤더와 가변 페이로드로 구성된다.

크기 필드 설명
1 byte타입 식별자항상 9. PonsCast 프레임임을 나타냄
4 bytes시퀀스 번호uint32, 프레임 생성 순서. 순서 맞추기와 누락 감지에 사용
8 bytes타임스탬프float64, Date.now() 값. jitter buffer 대기 시간 계산에 사용
N bytes페이로드실제 미디어 데이터 (비디오/오디오 청크)
💡 타입 식별자가 필요한 이유: DataChannel은 채팅, 제어, 자막 등이 같은 채널을 공유할 수 있다. 타입 식별자가 있으면 수신자가 "이 데이터가 PonsCast인지, 채팅인지"를 빠르게 판단할 수 있다.

🖥️ 발신자: 미디어 캡처부터 전송까지

발신자는 두 가지 캡처 경로를 갖는다. 첫 번째 경로는 captureStream()이다. 비디오 요소에서 MediaStream을 추출하고, MediaRecorder로 인코딩한 뒤 청크를 받는다. 데스크톱과 Android에서 주로 사용된다.

두 번째 경로는 Canvas 폴백이다. captureStream()이 지원되지 않는 환경(구형 브라우저)에서 비디오 프레임을 Canvas에 그린 뒤 toBlob()으로 추출한다. 성능은 떨어지지만 동작은 보장된다.

캡처된 청크는 wrapChunk() 함수를 거친다. 이 함수가 13바이트 프레임 헤더를 앞에 붙인다. 시퀀스 번호는 1부터 시작해서 청크가 생성될 때마다 1씩 증가한다.

프레임이 만들어지면 createBroadcaster()가 처리한다. 이 모듈은 토큰 버킷 알고리즘으로 속도를 제한한다. 최대 전송 속도는 초당 약 6MB, 버스트는 256KB, 틱 간격은 16ms다.

백프레셔가 작동한다. 발신자는 DataChannel의 bufferedAmount를 주시하다가 512KB를 넘으면 전송을 멈추고, 256KB 이하로 내려가면 다시 시작한다. TCP의 흐름 제어와 같은 원리다.

📺 수신자: MediaSource로 실시간 재생하기

수신자는 MediaSource Extensions(MSE)를 사용한다. 이 API는 브라우저가 미디어 데이터를 실시간으로 받아 재생할 수 있게 해주는 표준이다. 먼저 MediaSource 객체를 만들고, SourceBuffer를 추가하고, 비디오 요소의 src에 object URL을 연결한다.

중요한 것은 jitter buffer다. 기본값은 1500ms. 프레임이 도착하면 즉시 재생하지 않고, 잠시 대기시킨다. 네트워크에서는 프레임이 순서대로 오지 않을 수 있기 때문이다. 1.5초 동안 기다리면 늦게 온 프레임이 빈자리를 채울 수 있다.

시퀀스 번호 기반 순서 맞추기도 중요하다. 수신자는 packetQueue라는 Map을 가지고 있고, 키는 시퀀스 번호다. 새 프레임이 오면 해당 시퀀스 번호 위치에 넣는다. 다음에 재생할 시퀀스(nextSeq)가 준비되면 SourceBuffer에 appendBuffer한다.

누락 패킷 처리도 시퀀스 번호로 이뤄진다. nextSeq에 해당하는 프레임이 오지 않고, jitter buffer 시간이 초과되었거나, 시퀀스 번호 차이가 10을 넘으면 해당 프레임은 건너뛴다. "완벽한 전송"이 아니라 "충분히 빠른 전송"을 목표로 한다.

큐 오버플로 방지도 있다. packetQueue에 프레임이 30개 이상 쌓이면, 가장 최신 시퀀스에서 5개 뒤의 것들을 전부 버린다. 늦게 온 프레임은 새 프레임보다 가치가 낮기 때문이다.

🎵 코덱 협상: 어떤 포맷으로 재생할 것인가

수신자는 브라우저가 지원하는 코덱을 탐색한다. getSafePonsCastMimeType() 함수가 후보 코덱 목록을 순서대로 검사한다.

우선순위 MIME 타입
1요청된 MIME 타입
2video/webm;codecs=vp8,opus
3video/webm;codecs=vp9,opus
4video/webm
5video/mp4;codecs=avc1,mp4a
6video/mp4

MediaSource.isTypeSupported()로 각 후보를 테스트해서, 처음으로 지원되는 것을 선택한다. VP8이 먼저 오는 이유는 WebM이 MSE에서 지원 범위가 넓기 때문이다.

⚖️ 왜 화면 공유보다 나은가

플랫폼 독립성. 화면 공유 API를 쓰지 않기 때문에 iOS Safari에서도 동작한다. 브라우저가 MSE를 지원하기만 하면 된다.

제어 권한. 발신자가 "이 영상만 보여주고 싶다"고 할 수 있다. 화면 공유처럼 전체 화면이 노출되지 않는다.

메타데이터 제어. 발신자는 스트리밍 시작/끝 시점, 파일 이름, MIME 타입 등을 메타데이터로 보낼 수 있다. 수신자는 이 정보로 UI를 구성하고, 재생 상태를 동기화한다.

단점도 있다. 인코딩 비용이 발신자 쪽에서 발생하고, 패킷 손실 시 복구 메커니즘이 없다. lost packet은 그냥 건너뛴다. 하지만 PonsLink는 이 단점들을 백프레셔와 jitter buffer로 완화한다.
