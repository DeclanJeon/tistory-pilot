사용자에게는 “입장” 버튼 하나다. 하지만 PonsLink 쪽 코드를 따라가면 그 버튼은 꽤 많은 조건을 통과해야 한다. URL이 맞는지, guest token이 있는지, local media가 준비됐는지, session identity가 있는지, 마지막으로 orchestrator에 넘길 roomParams가 완성됐는지 봐야 한다.

이 글은 그 버튼 뒤의 경로를 좁게 따라간다. 화면 전체를 설명하기보다 Lobby, RoomGate, Room.tsx, useRoomOrchestrator 사이에서 책임이 어떻게 넘어가는지만 본다.

먼저 URL이 정책을 만든다

PonsLink에는 /lobby/:roomTitle, /room/:roomTitle, /join/:roomTitle이 모두 있다. 같은 방처럼 보이지만 의미가 다르다. /join은 reservation/session access를 통과한 사용자의 경로에 가깝고, /room은 RoomGate가 public desk나 room type 정책을 먼저 판단한다.

경로사용자에게 보이는 의미코드에서 하는 일
/lobby/:roomTitle입장 전 대기/준비nickname, media, session 준비
/room/:roomTitle방 링크 접근RoomGate가 type과 정책을 먼저 본다
/join/:roomTitle예약/초대 기반 입장reservation token과 session access가 중요하다

Room.tsx는 거대한 화면이 아니라 조립자다

Room.tsx는 버튼과 패널만 가진 컴포넌트가 아니다. query token을 읽고, guest access가 필요한지 확인하고, session을 만들고, localStream이 준비됐는지 보며, 마지막에 roomParams를 만든다. 이 객체가 만들어지기 전까지 realtime runtime은 시작되지 않는다.

여기서 중요한 점
방 입장의 기준은 “URL을 열었다”가 아니다. roomTitle, sessionUserId, sessionNickname, localStream, guestAccessReady가 맞물려야 한다.

media와 session은 UI보다 먼저 준비된다

입장 경험에서 media permission은 UX 문제처럼 보이지만, 코드에서는 runtime prerequisite이다. local stream이 없으면 peer connection을 만들어도 보낼 track이 없다. session user id가 없으면 signaling과 peer map의 key가 흔들린다.

orchestrator로 넘어가는 순간

useRoomOrchestrator(roomParams)가 호출되면 역할이 바뀐다. 이제 React page는 준비를 끝냈고, signaling connection, peer 생성, data channel message routing, whiteboard/chat/caption fanout은 hook과 store 계층이 맡는다.

글을 쓰며 남긴 판단: Room.tsx를 “화면 컴포넌트”로만 보면 너무 크다. “입장 조건 조립자”로 보면 책임이 훨씬 선명해진다.
